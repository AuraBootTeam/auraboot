## 一、协议性质

### Q1. AuraBoot 是开源软件吗?
**是,但更准确的说法是 "source-available with commercial restrictions"**。AuraBoot 协议以 Apache License 2.0 为基底,叠加了若干商业用途限制(主要是禁止把 AuraBoot 本身作为低代码/AI 平台 SaaS 转售,详见 §5.4)。它**不是 OSI 认证的开源协议**,因为 OSI 定义不允许"限制特定使用领域"。

社区里类似定位的项目:Sentry、PostHog、Cal.com、n8n、Cockroach、HashiCorp。

### Q2. 为什么不直接用 Apache-2.0 / MIT / AGPL?
- **纯 Apache/MIT** 防不住云厂商把 AuraBoot 包装成托管服务卖钱,同时不回馈
- **AGPL** 在国内大企业法务那里通过率低,会拦住正常自部署客户
- **本协议** 的目标:对个人 / 内部使用 / ISV 项目交付 完全友好;只对"把 AuraBoot 本身当低代码平台 SaaS 卖"的形态收费

### Q3. 协议会突然变更吗?
未来版本可能修订,但**已发布版本对旧版本号代码永久有效**。新版本只对新发布的代码生效。这是社区信任的基线,我们不会做 ElasticSearch / Redis 那种事后追溯改 license。

## 二、自部署 / 内部使用

### Q4. 公司内部部署 AuraBoot 给员工用,需要付费吗?
**不需要。** 商业用途允许(§5.1)。改源码也允许,只要保留版权声明(§5.3)和品牌信息(§5.2)。

### Q5. 我可以修改源码吗?要把改动开源吗?
**可以修改,且不强制开源你的改动。** 无论是内部部署还是对外交付,本协议都不要求你公开源码 — 这一点和 AGPL/GPL 完全不同。

你的改动可以保持私有,即使部署给客户也不需要附源码。我们鼓励但不强制把通用 bug 修复 / 改进回流到上游(via PR)。

### Q6. 给客户做项目交付,部署到客户机房,算分发吗?
**算分发,但完全允许,且不强制开源改动。** 你需要做的只有:
- 保留 LICENSE.txt 和版权声明(§5.1, §5.3)
- 保留 AuraBoot 品牌标识,左上角主 LOGO 可换成自己的(§5.2)
- 不把它转售为低代码/AI 平台 SaaS 服务(§5.4)

你对源码的修改、你的业务代码、你的插件 — **都可以保持闭源**。这种 ISV / 系统集成商场景是协议明确支持且友好的形态。

### Q7. 我可以 fork 一份私有仓库长期维护吗?
**可以。** 私有 fork 不算分发。只要你不对外发布这个 fork,完全没有义务公开源码。

## 三、品牌与定制化

### Q8. 我能改掉 AuraBoot 的 logo 和名字吗?
- **左上角主 LOGO**:可以换成你自己的(§5.2 例外)
- **页脚 "Powered by AuraBoot"、关于页、版权声明**:必须保留
- **代码里的版权头、license 头、署名注释**:必须保留(§5.3)
- **完全去品牌(white-label)**:需要购买商业 License

### Q9. 我做的产品基于 AuraBoot,但客户看不到 AuraBoot 字样可以吗?
如果你只是用 AuraBoot 配置出业务应用(ERP / CRM / 内部系统),终端用户主要看到的是**你的业务界面**,这是允许的。但管理后台 / 配置界面 / 关于页等暴露 AuraBoot 平台层的地方,品牌必须保留,除非买商业 License。

## 四、SaaS / 平台服务(关键边界)

### Q10. 什么样的服务算 §5.4 禁止的"低代码/AI 平台 SaaS"?
**判断标准:你卖的是"业务应用"还是"低代码能力本身"?**

| 场景 | 是否允许 |
|---|---|
| 你做一个 ERP SaaS,底层用 AuraBoot 跑业务,客户用的是 ERP 功能 | ✅ 允许(社区版即可) |
| 你做一个项目管理 SaaS,客户用项目管理功能 | ✅ 允许 |
| 你做一个"在线表单 / 在线低代码平台"SaaS,让客户来配置自己的应用 | ❌ 禁止,需商业 License |
| 你提供 "AuraBoot 托管服务",客户登录后看到的就是 AuraBoot 的配置界面 | ❌ 禁止 |
| 你把 AuraBoot 的 API 包装成"应用搭建 API"对外卖 | ❌ 禁止 |
| 你给单个客户私有部署一个低代码平台(单租户、客户自有) | ✅ 允许(算 ISV 交付) |
| 多个客户共用一个 AuraBoot 实例,各自配置应用 | ❌ 多租户 SaaS,禁止 |

**核心**:"主要价值主张是平台开发能力 vs 具体业务应用"是分界线(§7.4)。

### Q11. 我做了一个"垂直行业 SaaS"(比如餐饮 SaaS、教育 SaaS),底层用 AuraBoot,允许吗?
**允许,即使是多租户。** 因为你卖的是"餐饮/教育业务应用",不是"通用低代码平台"。租户用的是你预配置好的业务功能,而不是来"自己搭应用"。

如果你的产品同时提供"租户自己用低代码改业务流程"的能力,边界开始模糊 — 建议联系 license@auraboot.com 确认。

### Q12. 商业 License 能解锁多租户低代码 SaaS 吗?
**部分可以**,但 §7.3 明确即使买了商业 License,**也不能在未经书面同意下把 AuraBoot 改造成竞品低代码平台对外销售**。这是防止商业客户绕道做同业竞争。如果你有这种需求,需要单独谈 OEM 协议。

## 五、开发与贡献

### Q13. 我提交 PR 需要签什么协议?
需要签 **CLA(Contributor License Agreement)**。CLA 把你的代码版权 grant 给 The AuraBoot Project,这样我们才能在双协议(社区 + 商业)下合法分发。

CLA 不要求你放弃版权,只是授予项目长期、不可撤销的使用许可。第一次 PR 时 CLA bot 会引导签署。

### Q14. 我的 PR 会被用到商业版吗?
**会。** 这正是 CLA 的目的之一。你贡献的代码会同时出现在社区版和商业版里。如果你不接受这一点,请不要提交 PR。

### Q15. 我可以基于 AuraBoot 开发并销售插件吗?
**可以。** 插件是独立作品,你可以选择任何 license(包括闭源商业 license)。AuraBoot 协议不传染到插件层。

但如果你的插件**直接复制了 AuraBoot 核心代码**,那一部分代码仍受本协议约束。建议插件通过我们的扩展点 / SPI / DSL 接入,不要直接 fork 核心代码。

## 六、商业 License

### Q16. 什么情况下我必须买商业 License?
- ✅ 想去除全部品牌信息(white-label)
- ✅ 想做多租户低代码 / AI 平台 SaaS 转售
- ✅ 公司法务不接受非 OSI 认证的协议(罕见但存在)
- ✅ 想要官方技术支持 / SLA / 优先 bug 修复
- ✅ 需要定制开发 / 私有功能 / proprietary plugin

仅自部署、仅做项目交付、仅做垂直业务 SaaS — **都不需要商业 License**。

### Q17. 商业 License 怎么获取?
邮件 license@auraboot.com 或访问 https://www.auraboot.com/。

### Q18. 商业 License 是 perpetual 还是订阅?
具体条款以双方签订的商业合同为准。一般有 perpetual buy-out 和 annual subscription 两种形态可选,后者包含升级和支持。

## 七、合规与执行

### Q19. 如果我违反了协议会怎样?
§6.1:协议自动终止,你有 30 天时间整改。整改后协议恢复;30 天内未整改,你**必须停止使用并销毁所有副本**。如果违规造成商业损失(比如未经授权 SaaS 转售),Licensor 保留追究法律责任的权利。

### Q20. 协议受哪国法律管辖?
中国法律(§8.1),争议提交 CIETAC(中国国际经济贸易仲裁委员会)在北京仲裁,仲裁语言中英双语(§8.2)。

### Q21. 我所在国家不接受中国管辖怎么办?
对于跨境商业客户,我们可以在签订商业合同时另行约定管辖地(如新加坡 / 香港 / 客户所在地)。社区版用户默认按 §8.1 处理。

## 八、和其他项目对比

### Q22. AuraBoot 协议和 Appsmith / ToolJet / Budibase 比有什么区别?

| 项目 | 协议 | 多租户 SaaS 限制 | 品牌可改 |
|---|---|---|---|
| **AuraBoot** | Apache-2.0 + 补充条款 | 禁止低代码/AI 平台 SaaS 转售 | 仅左上 LOGO |
| Appsmith | Apache-2.0 | 无限制 | 完全可改 |
| ToolJet | AGPLv3 | 必须开源全部改动 | 完全可改 |
| Budibase | GPL v3 + Commercial | 必须开源全部改动(GPL) | 商业版可改 |

**总结**:AuraBoot 在"自部署 + ISV 项目交付 + 垂直 SaaS"场景下比 AGPL/GPL 系更友好(不强制开源你的业务代码),但比纯 Apache 系多了一道"防云厂商白嫖"的护栏。
